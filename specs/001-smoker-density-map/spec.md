# Feature Specification: 菸槍熱點地圖 (Smoker Density Map)

**Feature Branch**: `001-smoker-density-map`

**Created**: 2026-06-19

**Status**: Draft

**Input**: User description: "我想要做一個專案，是有地圖可以標記台灣現在哪裡菸槍最多，不喜歡煙味的人就可以不要靠近那裏，網友可以自行上來標記地點"

## Clarifications

### Session 2026-06-19

- Q: SourceHash 實際上是由什麼資料雜湊而成？ → A: 裝置指紋（User-Agent + 靜態瀏覽器屬性，不含 IP），個資法風險最低
- Q: 是否在 UI 加入免責聲明以降低侵權主張風險？ → A: 頁面底部固定免責文字 + 送出回報前確認彈窗
- Q: 是否揭露 Cloudflare Turnstile 的資料傳輸以符合《個資法》告知義務？ → A: 獨立一頁輕量隱私政策，僅列 Turnstile 為第三方資料處理者
- Q: 是否在回報中區分「合法吸菸區」與「疑似違規」以避免對場所業者的誤解？ → A: 不區分類型，Assumptions 補充回報反映煙味體驗而非法律合規評定
- Q: 是否提供資料刪除申訴管道以因應《個資法》主張？ → A: 隱私政策頁列出電子郵件申訴聯絡方式，人工處理
- Q: 兩筆回報要多近才算「同一地點」（聚合與去重的空間粒度）？ → A: 約 100 公尺格子（街區級解析度）
- Q: 登入會員相較匿名多了什麼能力？ → A: 會員可檢視與刪除自己提交的回報（匿名回報送出後不可追蹤）
- Q: 反饋要達多少才壓低某熱點？ → A: 來自 3 個不同來源（SourceHash）的有效反饋
- Q: 如何避免遠端亂回報？是否要求人在現場附近？ → A: 軟性鄰近檢查——有定位且距所選地點過遠時仍可送出，但標記為「未驗證鄰近」且密度權重較低
- Q: 是否限制單一來源的整體投稿量？ → A: 設每來源每日回報上限（預設約 20 筆/日），超過則拒絕
- Q: 匿名身分易重置，如何擋機器人/洗點？ → A: 匿名送出需通過人機驗證（採免費方案如 Cloudflare Turnstile），登入會員免驗證

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 在地圖上查看菸味熱點 (Priority: P1)

不喜歡煙味的使用者打開應用，看到一張台灣地圖，上面以視覺方式（標記點／熱區顏色）顯示各地的菸味回報密度，讓他在出門或選擇路線時可以避開菸味最重的地方。

**Why this priority**: 這是整個產品存在的理由——讓怕煙味的人「能避開」。即使沒有任何新增標記功能，只要能呈現既有資料，產品就已對使用者有價值，是最小可行產品（MVP）的核心。

**Independent Test**: 在資料庫預先放入數筆地點回報後，開啟地圖即可看到對應位置的標記與密度呈現，且可縮放、平移瀏覽台灣各區域，無需任何登入或新增動作即可完成測試。

**Acceptance Scenarios**:

1. **Given** 系統中已有多筆地點回報，**When** 使用者開啟地圖，**Then** 地圖以台灣為中心顯示，且每個有回報的地點以標記或熱區呈現
2. **Given** 使用者在地圖上，**When** 放大某一區域，**Then** 該區域的菸味密度以更細的顆粒度呈現（例如個別標記或聚合數字）
3. **Given** 使用者點擊某個標記，**When** 標記被選取，**Then** 顯示該地點的回報次數與最近回報時間

---

### User Story 2 - 網友新增菸味地點標記 (Priority: P2)

任何網友看到某處菸槍很多，可以在地圖上選定位置、送出一筆「這裡菸味很重」的標記，讓資料持續累積、反映現況。

**Why this priority**: 沒有持續的群眾回報，地圖資料會過時失準。此功能讓資料自我更新，是產品長期有用的關鍵，但在已有初始資料的情況下並非第一天上線的絕對必要。

**Independent Test**: 使用者在地圖上點選或搜尋一個位置、送出標記後，重新整理地圖即可看到該位置出現新的回報並反映在密度上。

**Acceptance Scenarios**:

1. **Given** 使用者在地圖上，**When** 點選地圖上某個座標並送出標記，**Then** 該筆回報被儲存且立即反映在地圖密度上
2. **Given** 使用者送出標記，**When** 未填寫必要資訊或座標落在台灣範圍外，**Then** 系統拒絕並提示原因
3. **Given** 同一地點已有他人標記，**When** 使用者再次標記同一地點，**Then** 系統累加該地點的回報密度而非建立完全無關的孤立點

---

### User Story 3 - 標記時效與自然衰退 (Priority: P3)

菸味熱點會隨時間改變。較舊的回報在地圖上的權重逐漸下降，讓地圖反映「現在」而非歷史累積，避免某地點因一次性事件被永久標記為熱區。

**Why this priority**: 提升資料的「現時性」準確度，是體驗優化而非核心功能；MVP 可先以單純累計呈現。

**Independent Test**: 放入一筆數週前的回報與一筆當天的回報，確認地圖上呈現的密度權重明顯不同（舊的較淡／較低）。

**Acceptance Scenarios**:

1. **Given** 某地點僅有較舊的回報，**When** 使用者查看地圖，**Then** 該地點的熱度低於有近期回報的地點
2. **Given** 一筆回報已超過時效門檻，**When** 計算地圖密度，**Then** 該筆回報對熱度的貢獻降至最低或不再計入

---

### Edge Cases

- 使用者在台灣地理範圍外點選座標時如何處理？（拒絕並提示）
- 同一使用者短時間內對同一地點重複大量送出標記（洗點／灌水）如何防止資料失真？
- 某地點被惡意標記但實際無菸味，是否提供他人「回報不實／否決」的機制？
- 地圖上某區域回報極度密集時，標記如何聚合避免畫面雜亂與效能下降？
- 使用者裝置無法或不願提供定位時，是否仍能手動選點？（應可）
- 沒有任何回報資料時，地圖呈現何種空狀態提示？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統必須顯示一張以台灣為預設範圍的互動式地圖，支援縮放與平移。
- **FR-002**: 系統必須將既有的菸味地點回報以視覺方式（標記點及／或密度熱區）呈現在地圖對應座標上。
- **FR-003**: 使用者必須能夠在地圖上選定一個位置並送出一筆菸味回報，無需註冊帳號即可進行（匿名回報）。
- **FR-004**: 系統必須驗證送出的座標位於台灣（含離島）合理地理範圍內，否則拒絕並提示。
- **FR-005**: 系統必須在同一地點累計多筆回報，並以回報量反映該地點的菸味密度高低。
- **FR-006**: 使用者必須能夠點選任一標記，查看該地點的回報次數與最近一次回報時間。
- **FR-007**: 系統必須在地圖區域回報密集時，將鄰近標記聚合呈現以維持可讀性。
- **FR-008**: 系統必須對單一來源在短時間內針對同一地點的重複回報加以節流或去重，以降低灌水對密度的影響。
- **FR-009**: 系統必須讓菸味回報具有時間維度，使較新的回報能比較舊的回報更高地反映於地圖密度（隨時間自然衰退）。
- **FR-010**: 系統必須提供回報不實或地點已改善的反饋管道，使社群可修正錯誤或過時的標記。
- **FR-011**: 系統必須在無任何回報資料時，向使用者顯示明確的空狀態提示。
- **FR-012**: 系統必須允許使用者選擇使用裝置定位或手動於地圖選點來指定回報位置。
- **FR-013**: 系統必須讓登入會員檢視並刪除自己提交的回報；匿名回報送出後不提供使用者追蹤或刪除（此為會員相較匿名的差異化能力）。
- **FR-014**: 當使用者有提供裝置定位時，系統必須比較定位與所選回報地點的距離；過遠（超過鄰近門檻）的回報仍可送出，但須標記為「未驗證鄰近」並以較低密度權重計入。未提供定位的回報視同未驗證鄰近。
- **FR-015**: 系統必須限制單一來源（SourceHash）每日可送出的回報總數（預設上限約 20 筆/日，可調），超過上限時拒絕並提示，以防止跨地點大量灌水。
- **FR-016**: 匿名回報送出前必須通過人機驗證（CAPTCHA，採免費服務）；登入會員免驗證。驗證失敗則拒絕回報，以阻擋自動化腳本繞過去重與每日上限。
- **FR-017**: 應用必須於所有頁面底部固定顯示免責聲明，說明回報內容為使用者主觀眾包意見、非官方或客觀事實陳述，平台不為回報內容之準確性負責。
- **FR-018**: 使用者送出回報前，系統必須顯示確認彈窗，重申回報為個人主觀觀察並請使用者確認，以建立知情同意紀錄、降低平台侵權責任風險。
- **FR-019**: 系統必須提供獨立靜態隱私政策頁面，揭露 Cloudflare Turnstile 為第三方資料處理者及其資料收集目的（人機驗證），以符合台灣《個人資料保護法》第 8 條告知義務；並提供電子郵件申訴聯絡方式，供使用者就資料刪除或不當回報提出申訴，由人工審查處理。

### Key Entities *(include if feature involves data)*

- **菸味回報 (Smoke Report)**: 代表一筆網友提交的觀察。關鍵屬性：地理座標（經緯度）、提交時間、可選的來源識別 SourceHash（用於去重／節流，非個人身分；由裝置指紋派生，即 User-Agent 與靜態瀏覽器屬性的雜湊，不含 IP 位址，以規避台灣《個人資料保護法》適用）、鄰近驗證狀態（依 FR-014 是否經定位確認在現場附近，影響密度權重）。
- **地點熱點 (Hotspot)**: 由鄰近的多筆回報聚合而成的區域或地點，代表菸味密度。聚合與去重以約 100 公尺的格子（街區級）為「同一地點」單位。關鍵屬性：代表座標／範圍、聚合後的密度權重、回報總數、最近回報時間。
- **反饋 (Feedback)**: 對既有標記的修正性回應，例如「此處已無菸味」或「回報不實」。當同一格收到來自 3 個不同來源（SourceHash）的有效反饋時，壓低該格熱度。關鍵屬性：對應的地點、反饋類型、來源識別、提交時間。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 使用者開啟應用後，地圖與既有菸味密度在 3 秒內完成呈現。
- **SC-002**: 使用者從決定標記到完成送出一筆回報，可在 30 秒內、3 個操作步驟內完成。
- **SC-003**: 新送出的回報在 5 秒內反映於地圖密度呈現中。
- **SC-004**: 在單一區域聚集數百筆標記時，地圖仍能順暢縮放與平移而無明顯卡頓。
- **SC-005**: 首次使用者中至少 80% 能在無說明的情況下，獨力找到並避開地圖上菸味最重的地點。
- **SC-006**: 透過去重／節流，單一來源的重複灌水對任一地點密度的影響不超過合理單筆權重，使密度仍能反映真實多人回報。

## Assumptions

- 服務範圍限定台灣（含離島），不處理海外地點。
- 第一版採匿名回報，不要求使用者註冊或登入；來源識別 SourceHash 由裝置指紋（UA + 靜態瀏覽器屬性，不含 IP）派生，做去重與節流，不屬台灣《個人資料保護法》所定義之個人資料，無需申報蒐集。
- 「菸槍最多」以群眾主觀回報的密度近似呈現，並非客觀空氣品質量測；產品定位為社群眾包指引而非官方數據。回報反映使用者的煙味體驗感受，不代表對特定地點或業者之法律合規狀態的評定，亦不區分合法吸菸區與疑似違規地點。
- 行動裝置瀏覽為主要使用情境，但桌面瀏覽亦受支援。
- 時效衰退採固定的時間門檻／權重函式，門檻值於規劃階段決定，可後續調整。
- 反饋機制以輕量否決／更新為主，第一版不引入完整審核後台或人工審查流程。
- 不在本規格範圍內：使用者個人帳號系統、推播通知、路線規劃導航、跨國擴充。
